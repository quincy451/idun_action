12 × 21 fat pixels

320 × 200 pixels

About your “2×2 to 320×200 BITMAP” idea
This is where I would be careful.
On a modern PC, a 2×2 bitmap is simple.
On a C64, a 2×2 high-res bitmap piece is not really independent, because the colors belong to the surrounding 8×8 cell, not to the individual pixels.
So if you allow:
BITMAP ExplosionPiece size 16,16
you have two choices.
Option 1: Make small bitmaps cell-aligned
This is the easiest and cleanest.
For high-res bitmap fragments:
width must be multiple of 8height should be multiple of 8
For multicolor bitmap fragments:
logical width must be multiple of 4height should be multiple of 8
Then the asset can carry its own cell color data cleanly.
Example:
BITMAP Logo 64,32MBITMAP Face 40,32
These are easy to store, load, blit, and validate.
Option 2: Allow arbitrary sizes but treat them as raw pixel masks
This gives you:
BITMAP Spark 2,2
But then the color rules are not self-contained. When you draw that 2×2 image into the screen, its colors have to merge with the destination 8×8 cell’s existing colors.
That can cause problems:
Destination cell already uses red and black.Your 2×2 image wants yellow.
In high-res mode there is no third color slot. So the compiler/runtime must either reject it, remap it, or replace one of the existing cell colors.
For that reason, I would probably use different names:
BITMAP      full-screen or cell-aligned high-res bitmapMBITMAP     full-screen or cell-aligned multicolor bitmapBMASK       arbitrary-size 1-bit maskMCMASK      arbitrary-size multicolor maskCHAR        8×8 character/tileMCHAR       4×8 multicolor character/tile
That keeps the language honest.

What happens if the user picks an illegal color?
The important thing is this:
The C64 does not store “illegal colors” in the bitmap data.
The VIC-II only sees bit patterns and color slots.
For high-res bitmap mode, every pixel is just:
0 or 1
The current 8×8 cell says:
0 = this color1 = that color
So if the user paints a third color into the same 8×8 cell, the hardware cannot represent that intention.
The editor/compiler has to decide what to do.
Example: high-res bitmap cell
Suppose one 8×8 cell currently uses:
slot 0 = blackslot 1 = white
The pixel data only stores:
0 = black1 = white
Now the user selects red and paints inside that same 8×8 cell.
The C64 cannot store:
0 = black1 = white2 = red
There is no 2.
So your editor has four possible behaviors.
Behavior 1: Reject the color
This is the safest for a C64-native editor.
Beep / flash border / show message:"Cell already uses black and white. Replace a color?"
This prevents accidental corruption.
Behavior 2: Replace one of the cell colors
The editor could ask:
This cell already uses:0 = black1 = whiteReplace which color with red?
If the user replaces white with red, then every white pixel in that 8×8 cell becomes red.
That is exactly how the C64 would behave, because all pixels using slot 1 now point to red.
Behavior 3: Auto-map to nearest existing color
If the user picks red but the cell only has black and white, the editor could choose the closest existing color.
That means red might become black or white.
This is useful for importing PC images, but annoying for hand drawing.
Behavior 4: Allow invalid art internally, then fail on export
This is the worst choice for your C64 editor/language.
The user draws something beautiful, then the compiler/exporter says:
Cannot export: cell 12,8 has 5 colors.
That is fine for a PC conversion tool, but bad for a C64-native art editor.

Best behavior for your C64 editor
For hand editing on the C64, I would use this rule:
Never allow the user to unknowingly create an invalid cell.
When the selected color would violate the cell rule, show a small cell-color panel on the right:
Current cell colors:High-res bitmap:  Slot 0: black  Slot 1: whiteSelected color:  redOptions:  A = replace slot 0  B = replace slot 1  C = cancel
For multicolor bitmap:
Current cell colors:Slot 0: global backgroundSlot 1: cell color ASlot 2: cell color BSlot 3: cell color CSelected color:  yellowOptions:  replace slot 1  replace slot 2  replace slot 3  cancel
Slot 0 in multicolor bitmap is special because it is usually the global background color. If you change that, it affects the whole screen, not just the current cell.
So I would protect it:
Changing background color affects entire image. Continue?

What the C64 actually displays
If the asset is forced into C64 format, it will always display “correctly” from the VIC-II’s point of view.
But it may not display as the artist intended.
For example, if a PC image has this in one 8×8 high-res cell:
blackwhiteredblue
When converted to C64 high-res bitmap, the exporter must reduce it to:
only 2 colors
Maybe it chooses black and white. Then red and blue pixels get mapped to black or white.
The VIC-II does not glitch or crash. It simply displays whatever the bits and color attributes say.
So the rule is:
The hardware does not handle illegal colors.The editor/compiler must handle them before the data reaches hardware.

The biggest design rule
For sprites:
The color limits belong to the sprite.
For bitmap graphics:
The color limits belong to each 8×8 cell.
So your editor should think in terms of current cell color slots, not just selected palette color.
For a C64-native editor, the right-side panel should probably show:
Global C64 palette: 16 colorsCurrent object colors:  sprite color  multicolor 1  multicolor 2orCurrent bitmap cell colors:  slot 0  slot 1  slot 2  slot 3
Then when the user paints, they are really painting with a slot, and the slot has a C64 color assigned to it.
That avoids most confusion.
So instead of the user thinking:
I am painting red.
They think:
I am painting with cell color slot 1, which is currently red.
That maps much more cleanly to the C64 hardware.

I’ll frame this as language/API design rather than raw VIC-II register trivia: what should be natural on a SPRITE/MSPRITE variable, and what should stay global because the hardware makes it shared.
Thought for 39s
For your language, I would split this into two concepts:
SPRITEDATA / MSPRITEDATA = the 63-byte picture definitionSPRITE / MSPRITE         = a live hardware sprite object using one of the 8 VIC-II sprite slots
That keeps the language clean. The C64 sprite image is a 24×21 object stored as 63 bytes, and the live sprite is controlled through VIC-II registers for position, enable/disable, color, expansion, priority, multicolor mode, and collision flags. The visible hardware slots are limited to 8 sprites. 
Core operations that make sense
1. Attach sprite art to a hardware sprite
You need a way to say:
Player.image = PlayerStandingEnemy.image  = EnemyFrame1
or:
Player.setImage("Player.SPR")
Under the hood, this sets the sprite data pointer for the hardware slot. On the C64, the sprite pointers live in the last 8 bytes of the current screen memory block, normally $07F8 to $07FF if screen RAM is at $0400. 
Good operations:
sprite.setImage(spriteData)sprite.setFrame(frameNumber)sprite.nextFrame()sprite.animate(firstFrame, lastFrame, speed)
Animation is a very natural high-level operation because the hardware already expects a sprite slot to point at a 64-byte sprite block.

2. Show / hide
This maps directly to the sprite enable register.
Player.show()Player.hide()Player.visible = truePlayer.visible = false
The VIC-II has an enable bit for each sprite, so this is a perfect high-level operation. 

3. Position and movement
The obvious operations are:
Player.x = 100Player.y = 80Player.setPosition(100, 80)Player.move(2, 0)Player.moveTo(x, y)
C64 sprite X position is effectively 9-bit, because the low 8 bits are in the sprite X register and the 9th X bit for all sprites is collected in $D010. The Y position is an 8-bit register. 
I would hide that from the programmer. Let them write:
Player.x = 270
and have the compiler/runtime automatically handle the high X bit.
I would also allow off-screen positions:
Player.x = -24Player.x = 340
Even if the hardware register cannot directly hold negative values, the language can keep a logical signed X/Y and only display the sprite when it is in range. That makes scrolling and enemy spawning cleaner.

4. Color
For a high-res sprite:
Player.color = RED
That is simple: one visible color plus transparency.
For a multicolor sprite:
Enemy.color = GREEN
But be careful: this should mean the individual sprite color, not all three colors.
The two extra multicolor sprite colors are shared globally by all multicolor sprites:
SpriteSystem.multiColor1 = ORANGESpriteSystem.multiColor2 = BROWN
or:
setSpriteMultiColors(ORANGE, BROWN)
So for MSPRITE, I would support:
Enemy.color = GREENEnemy.show()
But I would not support this as if it were fully per-sprite:
Enemy.color1 = ORANGEEnemy.color2 = GREENEnemy.color3 = BROWN
because that lies about the hardware. Multicolor sprites have three visible colors plus transparency, but two of those visible colors are shared among all multicolor sprites. 

5. Size / expansion
The C64 can double a sprite horizontally, vertically, or both. It cannot arbitrary-scale a sprite.
So I would support:
Player.expandX = truePlayer.expandY = truePlayer.normalSize()Player.doubleWidth()Player.doubleHeight()Player.doubleSize()
Do not make this look like modern scaling:
Player.scale = 1.5Player.scaleX = 3
Better:
Player.widthMode  = NORMAL | DOUBLEPlayer.heightMode = NORMAL | DOUBLE
A high-res sprite is normally 24×21 screen pixels. With expansion it can display as 48×21, 24×42, or 48×42. The VIC-II doubles the pixels; it does not add resolution. 

6. Priority: in front of or behind background
The C64 supports sprite-to-background priority. So this makes sense:
Player.priority = FRONTPlayer.priority = BEHIND_BACKGROUND
or:
Player.inFront()Player.behindBackground()
But be careful with sprite-to-sprite priority. When sprites overlap each other, lower-numbered sprite slots appear in front of higher-numbered ones. So this:
Player.z = 100Enemy.z = 50
is only possible if your runtime changes hardware slot assignments.
Simpler version:
Player.slot = 0   ; appears in front of slot 1, 2, 3...Enemy.slot = 1
Advanced version:
Player.zOrder = 1Enemy.zOrder = 2SpriteSystem.sortByZ()
But that requires a manager that assigns hardware sprite slots.

7. Collision operations
The C64 has hardware collision registers for sprite-to-sprite and sprite-to-background collisions. Reading those registers clears them, so a high-level language should probably buffer collision state once per frame. 
Good high-level operations:
if Player.hitSprite() then ...if Player.hitBackground() then ...if Player.hit(Enemy) then ...
But there are two levels here.
Hardware-level:
Player.hitSprite()Player.hitBackground()
This tells you whether the sprite was involved in a collision.
Software-level:
Player.overlaps(Enemy)Player.boxOverlaps(Enemy)Player.pixelOverlaps(Enemy)
This is more exact and more predictable.
I would provide both:
Player.hardwareCollision()Player.backgroundCollision()Player.boxCollision(Enemy)Player.pixelCollision(Enemy)
The hardware collision registers tell you which sprite numbers were involved, but they are not a perfect game-logic system. They are useful, but many games still use software bounding-box collision for control and reliability.

Thought for 13s
For the non-sprite graphics types, I would design the operations around what the C64 can really change cheaply:
1. Whole resource operations2. Cell/color-slot operations3. Pixel/bit operations4. Draw/blit operations5. Display/load operations
The important C64 reality is that bitmap graphics are still organized around character-style blocks. The VIC-II has 320×200 high-res or 160×200 multicolor bitmap resolution, but bitmap data is organized through 8×8 / 4×8 tile-like areas, with color information coming from screen RAM and sometimes color RAM. 
1. BITMAP / high-res bitmap
I would use this for high-res bitmap graphics, either full-screen or cell-aligned rectangular pieces.
BITMAP TitleBITMAP LogoBITMAP Panel
A high-res C64 bitmap should expose operations based on 1-bit pixels plus 2 colors per 8×8 cell. High-res bitmap mode uses screen RAM to hold foreground/background colors for each 8×8 tile. 
Useful operations
Title.clear(color0, color1)Title.setPixel(x, y, slot)Title.getPixel(x, y)Title.setCellColors(cellX, cellY, color0, color1)Title.getCellColors(cellX, cellY)Title.drawLine(x1, y1, x2, y2, slot)Title.drawRect(x, y, w, h, slot)Title.fillRect(x, y, w, h, slot)Title.copyFrom(sourceBitmap, srcX, srcY, w, h, dstX, dstY)Title.blit(sourceBitmap, dstX, dstY, colorPolicy)Title.loadTo(bitmapAddress, screenAddress)Title.display()Title.save("Title.HBM")
For setPixel, I would not have the programmer set an arbitrary color directly. I would have them set a slot:
slot 0 = cell color 0slot 1 = cell color 1
So this is good:
Title.setPixel(10, 20, 1)
This is dangerous:
Title.setPixelColor(10, 20, RED)
because that might force a cell-color change and accidentally recolor other pixels in the same 8×8 cell.
If you do allow setPixelColor, it should need a policy:
Title.setPixelColor(x, y, RED, FAIL_IF_ILLEGAL)Title.setPixelColor(x, y, RED, REPLACE_CELL_COLOR)Title.setPixelColor(x, y, RED, NEAREST_COLOR)

I would not make this an RGB palette. I would make it a list of C64 color indexes.

The design rule I would use
For anything bitmap-like, expose slot-based drawing as the normal method:
bitmap.setPixel(x, y, slot)
Only expose direct color drawing with an explicit policy:
bitmap.setPixelColor(x, y, color, FAIL_IF_ILLEGAL)bitmap.setPixelColor(x, y, color, REPLACE_CELL_COLOR)bitmap.setPixelColor(x, y, color, MAP_TO_EXISTING)
For anything screen/tile-like, expose character-cell operations:
screen.putChar(x, y, charCode)screen.setColor(x, y, color)tile.drawTo(screen, x, y)
For the C64, this gives you a nice split:
Sprites      = moving objectsBitmaps      = full-screen/title/art panelsCharsets     = reusable pixel definitionsScreens      = 40×25 map of charactersTiles        = reusable screen-cell blocksMasks        = drawing/collision helpers
That maps well to both the editor and the compiler.
