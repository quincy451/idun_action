RUNTIME PROGRAMS SPEC

Purpose

This document defines the current runtime contract for ActionC64U programs.

The maintained product path no longer uses a separate runtime program. The
linker output is the program that runs.

Current model:

- ACTC.PRG compiles SRC/<MODULE>.ACT to OBJ/<MODULE>.OBJ.
- ALINK.PRG links OBJ/<MODULE>.OBJ and its dependency closure.
- ALINK.PRG emits BIN/<MODULE>.PRG as a directly launchable Commodore program.
- No separate runner is required after linking.

Execution Model

Build-time:

- ACTC.PRG parses source and writes project object files.
- ALINK.PRG resolves the object graph, dead-strips unused code/data, selects
  required runtime helper modules, and writes the final PRG image.

Run-time:

- UDOS launches the linked PRG.
- The linked PRG contains its own entry point, selected runtime helpers, and
  program payload.
- Control returns to UDOS through the normal resident/tool return path when the
  program exits.

There is no second launch stage where a generic runtime program interprets or
hosts the linked output. If a helper is required, ALINK must put that helper into
the final PRG or into an explicitly linked payload that the final PRG owns.

Required Runtime Files

Minimum runtime set after a successful link:

- UDOS system/resident environment
- BIN/<MODULE>.PRG

Not required at program launch:

- ACTC.PRG
- ALINK.PRG
- OBJ/ files
- SRC/ files
- loose runtime module source/object files
- a generic VM or bytecode runner

ALINK Responsibility

ALINK is responsible for 100% of what the presented program needs in order to
run. That includes:

- the PRG load address and entry path
- linked code and data layout
- selected runtime helpers
- unresolved external closure loading
- dead-strip decisions
- any startup/return code needed by the program

A linked program must not depend on a separate resident instruction-set runtime
or a separate interpreter-style launcher to make the program meaningful.

Runtime Helper Policy

Optional feature families such as REAL math, printing, graphics, SID/sprite,
joystick/mouse input, DBF, TCP, and 80-column support should be link-selected
helper modules.

The rule is simple: if a program uses a feature, ALINK selects and places the
needed helper code into the linked PRG. If a program does not use a feature, that
helper code is not paid for in the final image.

Recommended Operator View

For someone building and running ActionC64U programs, the tool responsibilities
are:

- ACTC.PRG: compile source to object
- ALINK.PRG: link object closure to a directly launchable PRG
- BIN/<MODULE>.PRG: run this final program under UDOS

Current Proven Status

The active proof chain is:

- ACTC produces OBJ/<MODULE>.OBJ.
- ALINK produces BIN/<MODULE>.PRG.
- UDOS launches BIN/<MODULE>.PRG directly.

The maintained runtime direction is direct PRG output only.
